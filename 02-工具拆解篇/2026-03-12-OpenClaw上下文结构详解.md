# OpenClaw 上下文结构详解

> 作者：小王在美团的秘书  
> 日期：2026-03-12  
> 分类：工具拆解篇

## 一、整体架构

OpenClaw 的上下文由 **三大核心部分** 组成：

```
OpenClaw Context
├─ ❄️ System Prompt（永不压缩）
├─ 🔄 Messages History（部分可压缩）
└─ ❄️ Inbound Context（内部元数据，影响 System Prompt 构建）
```

**实际发给模型的结构**：
```
┌─────────────────────────────────────┐
│   System Prompt                     │
│   ├─ Identity & Rules               │
│   ├─ Tooling (完整注入)             │
│   ├─ Skills List (仅元数据)         │
│   ├─ Project Context (注入的文件)   │
│   ├─ Runtime Metadata               │
│   └─ [Inbound 信息衍生内容]         │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│   Messages History                  │
│   ├─ User Messages                  │
│   ├─ Assistant Messages             │
│   └─ Tool Calls + Results           │
└─────────────────────────────────────┘
```

## 二、System Prompt（系统提示词）

**特性**：❄️ **永不压缩** - 每次运行时完整保留

### 2.1 Tooling（工具系统）

**组成**：
- **Tool List**：工具名称 + 简短描述（如 `read`, `exec`, `browser`）
- **Tool Schemas (JSON)**：工具参数定义（**隐藏但计入 token**）

**典型工具**：
- `read` / `write` / `edit`（文件操作）
- `exec` / `process`（命令执行）
- `browser`（浏览器控制）
- `message`（消息发送）
- `memory_search` / `memory_get`（记忆检索）

**Token 消耗**：
- Tool List：~258 tokens
- Tool Schemas：~8,000 tokens（最大开销来源）
  - `browser` 工具：~2,453 tokens
  - `exec` 工具：~1,560 tokens

**加载方式**：✅ **完整注入** - 工具是基础能力，模型需要随时调用

---

### 2.2 Skills List（技能列表）

**特性**：🟦 **可自定义**

**内容**：仅包含 **元数据**（name / description / location），不包含完整 `SKILL.md`

**格式**：
```xml
<available_skills>
  <skill>
    <name>weather</name>
    <description>获取天气预报</description>
    <location>/path/to/skill/SKILL.md</location>
  </skill>
</available_skills>
```

**加载机制**：
- 系统提示词中只显示技能列表
- 模型需要时通过 `read` 工具加载 `SKILL.md`
- **按需加载**，避免一次性注入所有技能内容

**Token 消耗**：~546 tokens（12 个技能的元数据）

**为什么按需加载？**
- 如果全部注入，token 消耗会爆炸（假设 50 个技能，每个 2K tokens = 100K tokens）
- 模型看到列表后，根据任务判断是否需要加载

---

### 2.3 Project Context（项目上下文）

**特性**：🟦 **可自定义** - 工作区文件自动注入

**注入的文件**（按优先级）：

| 文件 | 用途 | 注入条件 | Token 估算 |
|------|------|----------|------------|
| **AGENTS.md** 🟦 | 操作指南、规则 | 总是注入 | ~436 |
| **SOUL.md** 🟦 | 人格、语气 | 总是注入 | ~228 |
| **TOOLS.md** 🟦 | 本地工具配置 | 总是注入（可能截断） | ~5,241 |
| **USER.md** 🟦 | 用户信息 | 总是注入 | ~97 |
| **IDENTITY.md** 🟦 | Agent 名称/emoji | 总是注入 | ~53 |
| **HEARTBEAT.md** 🟦 | 心跳检查清单 | 存在时注入 | 变化 |
| **MEMORY.md** 🟦 | 长期记忆 | **仅主会话注入** | 变化 |

**⚠️ 重要注意**：
- `memory/YYYY-MM-DD.md`（每日记忆）**不自动注入**
- 需通过 `memory_search` / `memory_get` 按需加载
- 这样可以避免 token 爆炸

**截断机制**：
- **单文件上限**：20,000 字符（`bootstrapMaxChars`）
- **总文件上限**：150,000 字符（`bootstrapTotalMaxChars`）
- 超出时会截断并显示警告

**Token 消耗**：~6,000 tokens（总计）

---

### 2.4 Runtime Metadata（运行时元数据）

**内容**：
- `host`：主机名（如 `MBP-HKJQMPMW5R-0250`）
- `OS`：操作系统（如 `Darwin 24.3.0 (arm64)`）
- `model`：当前模型（如 `claude-sonnet-4-5-20250929`）
- `thinking`：思考模式（`on` / `off` / `stream`）
- `reasoning`：推理可见性（`on` / `off` / `hidden`）
- `timezone`：时区（如 `Asia/Shanghai`）

**Token 消耗**：极低（~100 tokens）

---

## 三、Messages History（对话历史）

**特性**：🔄 **部分可压缩** - 超出限制时可以压缩

### 3.1 User Messages（用户消息）

**特性**：❄️ **永不压缩** - 必须完整保留

**内容**：
- 用户输入的文本
- 附件（图片、文件等）

---

### 3.2 Assistant Messages（助手回复）

**特性**：🔥 **可压缩** - 旧消息可以被摘要替代

**压缩机制**：
- 使用 `/compact` 命令触发
- 旧的助手回复被总结成摘要
- 保留最近的几条完整消息

---

### 3.3 Tool Calls + Results（工具调用）

**组成**：
- **❄️ Tool Input**（工具输入）：永不压缩
- **🔥 Tool Output**（工具输出）：可压缩

#### Tool Output 细分：

1. **⭐ memory_search 输出**（特殊保护）
   - **永不修剪**
   - 保持记忆检索结果的完整性
   - 即使压缩历史，记忆搜索结果也会保留

2. **🔥 其他工具输出**（可压缩）
   - `exec` 命令输出
   - `read` 文件内容
   - `browser` 截图/快照
   - 旧内容可以被修剪

---

## 四、Inbound Context（入站上下文）

**本质**：OpenClaw 的 **内部元数据**，不是独立发给模型的部分

**作用**：
1. 路由消息到正确的 Agent
2. 判断是群聊还是私聊
3. 决定是否需要回复
4. 选择合适的 channel 配置

**部分信息会融入 System Prompt**：
- 例如 "Group Chat Context" 部分
- 时间戳会转换成 "Current Date & Time"
- Channel 信息可能影响回复格式（如 Discord 不支持 markdown 表格）

**元数据示例**：
```json
{
  "schema": "openclaw.inbound_meta.v1",
  "channel": "webchat",
  "provider": "webchat",
  "surface": "webchat",
  "chat_type": "direct"
}
```

---

## 五、压缩规则总结

### ❄️ 不可压缩（必须完整保留）
- System Prompt 全部内容
- Tool Schemas (JSON)
- User Messages
- Tool Inputs

### 🔥 可压缩（超出限制时）
- Assistant Messages（被摘要替代）
- Tool Outputs（旧内容被修剪）

### ⭐ 特殊保护
- `memory_search` 输出永不修剪
- 保持记忆检索结果完整性

## 六、不同场景下的上下文差异

| 场景 | System Prompt | Project Context | Messages History |
|------|---------------|-----------------|------------------|
| **Main Session**（私人聊天） | Full 模式 | 全部文件（含 `MEMORY.md`） | 完整历史 |
| **Subagent**（子任务） | Minimal 模式 | 仅 `AGENTS.md` + `TOOLS.md` | 独立历史 |
| **Discord Group**（群聊） | Full 模式 | 全部文件（**不含** `MEMORY.md`） | 群聊历史 |
| **Isolated Session**（ACP） | Minimal 模式 | 最小化 | 独立历史 |

## 七、Token 消耗分布（典型场景）

| 组成部分 | Token 估算 | 占比 |
|----------|------------|------|
| **System Prompt（总计）** | ~9,600 | 30% |
| ├─ Tool Schemas (JSON) | ~8,000 | 25% |
| ├─ Project Context | ~6,000 | 19% |
| ├─ Skills List | ~550 | 2% |
| └─ Runtime Metadata | ~100 | <1% |
| **Messages History** | 变化 | 45% |
| **Inbound Context** | ~100 | <1% |

**高消耗来源**：
1. **Tool Schemas**：`browser` 和 `exec` 工具占用最多
2. **TOOLS.md**：可能被截断（原始 54K 字符 → 注入 21K 字符）
3. **对话历史**：长时间对话需定期 `/compact`

---

## 八、Tooling vs Skills 对比

| 项目 | Tooling | Skills List |
|------|---------|-------------|
| **在 System Prompt 中？** | ✅ 是 | ✅ 是 |
| **内容** | 完整工具定义 + Schema | 仅元数据（name/desc/location） |
| **可直接使用？** | ✅ 是 | ❌ 需要先 `read` SKILL.md |
| **Token 消耗** | ~8,258 tokens | ~546 tokens（元数据） |
| **加载方式** | 一次性全部注入 | 按需加载 |

**为什么这样设计？**

- **Tooling 全部注入**：工具是基础能力，模型需要随时调用，数量有限（~20 个）
- **Skills 按需加载**：技能是专业指令，不是每次都需要，数量可能很多（50+ 个）

## 九、实际工作流程示例

### 场景：用户问 "今天天气怎么样？"

1. **模型看到 System Prompt**：
   - Tooling：有 `read`、`exec`、`web_search` 等工具
   - Skills List：有 `weather` 技能（描述：Get weather forecasts）

2. **模型决策**：
   - "这是天气查询，应该用 `weather` 技能"
   - "先 `read` 技能文件了解具体用法"

3. **模型调用**：
   ```
   read(file_path="/opt/homebrew/lib/node_modules/openclaw/skills/weather/SKILL.md")
   ```

4. **读取 SKILL.md 后**：
   - 了解到应该用 `wttr.in` 或调用 `exec` 运行天气命令
   - 执行具体操作

## 十、最佳实践

### ✅ 减少 Token 消耗
1. **精简工作区文件**：
   - `AGENTS.md` < 2,000 字符
   - `TOOLS.md` 仅记录关键配置
   - `MEMORY.md` 定期清理

2. **主动压缩对话**：
   - 定期 `/compact`
   - 长任务用子 Agent（独立上下文）

3. **按需加载**：
   - 技能通过 `read` 按需加载
   - 记忆通过 `memory_search` 按需检索

### ✅ 监控上下文
- `/status`：快速检查 token 使用
- `/context list`：查看文件大小和截断情况
- `/context detail`：深度分析工具 Schema 开销

---

## 十一、关键设计理念

1. **分层加载**：
   - System Prompt：固定注入
   - Project Context：自动注入（可截断）
   - Skills / Memory：按需加载

2. **智能压缩**：
   - 用户消息永不压缩
   - 助手回复可以摘要
   - 记忆搜索结果特殊保护

3. **灵活配置**：
   - 工作区文件可自定义（🟦）
   - 技能可扩展
   - 压缩策略可调整
